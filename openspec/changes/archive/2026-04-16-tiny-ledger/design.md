## Context

TinyLedger 面向 8–10 岁儿童，产品目标是「小而美」的财商启蒙：用极简逻辑承载储蓄、消费、目标与增值直觉，并用现代审美与克制动效建立正向反馈。**客户端实现固定为 Flutter（Dart）**，以单一代码库交付 iOS/Android；设计需在「教育模拟」与「真实金融」之间划清边界，并优先保证可理解性与家长信任。

## Goals / Non-Goals

**Goals:**

- 建立可扩展的核心领域模型：虚拟余额、流水、目标进度、增值演示参数彼此解耦。
- 提供可本地优先的数据路径，降低儿童使用门槛（弱网可用），并为未来云同步留接口。
- 将动效与视觉语言产品化：统一时长曲线、反馈层级与减弱动效策略。
- 在关键路径上嵌入守护策略：适龄提示、默认隐私友好、无诱导性商业化默认。
- 以 Flutter 为第一公民构建 UI 与动效，并保证 `MediaQuery.disableAnimations` 等系统无障碍信号被尊重。

**Non-Goals:**

- 不提供真实资金充值、提现、银行托管或与真实证券/基金交易链路对接。
- 不在首版实现复杂家庭多子账号协同与完整家长端产品（仅保留接口与占位流程）。
- 不追求「游戏化数值膨胀」或重度任务系统，避免偏离财商启蒙主线。
- 不在首版并行维护「Flutter + 原生双栈」作为同等优先级的主实现路径（避免重复交付与分叉测试）。

## Decisions

0. **Flutter 为唯一客户端技术栈**  
   - 使用 Flutter 构建跨平台 UI；领域与用例逻辑放在 **纯 Dart** 层（不依赖 `Widget`），Widget 层仅编排状态与动效。  
   - 状态管理首版推荐 **Riverpod** 或 **Bloc** 二选一（在首个迭代内锁定，避免混用多套范式）。  
   - 备选：iOS/Android 各一套原生 UI。未采纳原因：双端重复实现与视觉一致性成本高，与「小而美」迭代节奏冲突。

1. **领域内核与展示层分离**  
   - 账本、流水、目标、增值演示作为纯领域模块；UI 与动效通过适配层消费领域事件。  
   - 备选：页面内直接拼业务。未采纳原因：动效与交互迭代会污染领域逻辑。

2. **货币采用整数「分」或最小单位存储，界面按「元」格式化**  
   - 避免浮点误差，便于儿童理解「加/减」一致性。  
   - 备选：字符串金额。未采纳原因：运算与校验成本更高。

3. **增值能力以「可解释模拟」交付**  
   - 采用固定规则（如周期奖励、单利可视化）并显著标注「学习用」。  
   - 备选：接入真实利率或市场数据。未采纳原因：合规与认知负担过高。

4. **动效系统以设计令牌 + Flutter 动画原语封装**  
   - 使用 `ThemeExtension` 或集中常量承载时长与曲线；对装饰性动画统一接入「减弱动效」分支（尊重 `MediaQuery.disableAnimations` 与用户设置）。  
   - 备选：每屏自由发挥动画。未采纳原因：一致性差且易触发眩晕/分心。

5. **首版持久化默认本地（Flutter 侧 SQLite 优先），同步服务抽象为接口**  
   - 推荐使用 **Drift**（或 `sqflite` + 手写 DAO）存流水与目标；领域层仅依赖仓储抽象。  
   - 快速验证闭环；后续可替换为远端同步实现。  
   - 备选：首版即上云。未采纳原因：信任与合规节奏更慢。

## Risks / Trade-offs

- [儿童误解为真实理财] 增值与余额展示被误解为「真钱」→ 全链路文案与视觉标注「虚拟/学习」，关键操作二次确认。
- [动效过度干扰学习] 动效抢戏 → 提供减弱动效与「安静模式」预设。
- [家长信任不足] 数据与权限不透明 → 预留家长可见摘要与导出占位，并在守护规范中固定最小披露原则。
- [范围膨胀] 功能堆叠削弱「小而美」→ 以 MVP 切片发布：钱包流水 → 目标 → 增值演示 → 动效打磨。

## Migration Plan

- 阶段 0：领域模型与本地持久化打通，完成最小流水闭环。  
- 阶段 1：加入目标存钱罐与进度反馈。  
- 阶段 2：加入增值演示与教育说明。  
- 阶段 3：动效与视觉系统统一，做可用性与家长访谈迭代。  
- 回滚策略：任一阶段以特性开关禁用新增模块，保留核心记账与余额。

## Open Questions

- 家长协同首版做到「占位链接」还是「同局域网配对」级别？  
- 目标与流水是否允许负数调整（纠错）还是仅允许「冲正记录」？  
- 本地化与货币符号默认策略（中国大陆场景是否固定 CNY）？  
- 首版是否纳入 **Flutter Web/Desktop** 作为实验构建（若纳入，动画与存储降级策略如何统一）？
